# AGENTS.md — documentation

> Scope: the Docusaurus docs site under `docs/`. For repo-wide rules,
> architecture, and the Go backend, read `../AGENTS.md` first — it wins on
> any conflict. This file only covers **writing and verifying docs**.

The golden rule from the root file: **a user-facing change isn't done until
it's documented here.** New flag, HTTP route, plugin, or integration → a page
lands (or grows) under `docs/content/`.

---

## Stack

* Docusaurus 3 (TypeScript config). Site config: `docusaurus.config.ts`.
* All prose lives under `docs/content/` as Markdown/MDX. Anything outside
  `content/` (including this file) is **not** published.
* The sidebar is **auto-generated from the directory tree** (`sidebars.ts`,
  `autogenerated`). You do not hand-edit a sidebar — placing the file in the
  right folder *is* how it appears. Order within a folder is controlled by the
  `sidebar_position` frontmatter key.
* The API reference is **not** written by hand: it renders from
  `../api/openapi.yaml`. Touch a route there, not in Markdown.

---

## Where each kind of content goes

```
docs/content/
├── getting_started/   # install, docker, go quickstart, first login
├── general/           # concepts: alerts, rules, actions, conditions,
│   │                  #   clustering, inputs, notifications, query language…
│   └── integrations/  # ONE page per integration (input / output / webhook)
├── configuration/     # one page per file-config section
│   │                  #   (core, postgres, sqlite, ingest, ldap_auth,
│   │                  #    notifications, housekeeping, syncer, general)
└── migration/         # Python 1.x → Go upgrade notes
```

Decision table for "I added X, where does it document?":

| You added…                                   | Document in…                                                  |
|----------------------------------------------|---------------------------------------------------------------|
| A new integration (input daemon / notifier / `WebhookReceiver`) | `general/integrations/<name>.md`, and link it from `general/inputs.md` or `general/notifications.md` |
| A new file-config field / section            | the matching `configuration/<section>.md` (mirror the `schema/*.go` struct) |
| A new concept / pipeline behaviour           | a page under `general/`                                       |
| A new or changed HTTP route                  | `../api/openapi.yaml` (the API reference auto-renders it)     |
| A new install path / first-run step          | `getting_started/`                                            |

Keep `general/integrations/` in lockstep with the plugin set in
`../internal/pluginimpl/all/all.go` and the `cmd/snooze-*` input/output
daemons — every shipped integration should have exactly one page.

## Page conventions

* Frontmatter: set `sidebar_position` to order siblings; `title`/`id` only if
  you need to override the filename-derived defaults.
* Put images in the local `images/` folder next to the page; reference them
  with a relative path.
* Match the voice of neighbouring pages — operator-facing, example-first.
* Cross-link rather than duplicate (e.g. an integration page links to the
  relevant `configuration/` section instead of restating config).

## Verify before you call it done

```bash
task docs:install     # once, installs node deps (npm ci)
task docs:serve       # live preview on http://localhost:3000
task docs:build       # MUST pass — build fails on broken links / bad MDX
task docs:typecheck   # type-check config + custom components
```

`task docs:build` green (no broken-link warnings) is the bar. Don't hand-edit
`docs/build/` or `docs/.docusaurus/` — they're generated.
